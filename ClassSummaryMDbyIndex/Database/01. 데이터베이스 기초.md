# 데이터베이스 기초

___

## 1. 개요

___

### 데이터 저장의 필요성

- 사용자 정보, 게시글, 거래 내역 등 지속적으로 유지되어야 하는 데이터는 반드시 영구 보존이 필요
- 기존에는 파일 시스템을 활용해 영구 저장해 왔음
- 이후 데이터를 체계적으로 제어 및 관리하는 데이터베이스 관리 시스템(DBMS)이 등장

### 파일 시스템 방식의 한계

- 데이터 중복성: 동일한 내용의 데이터가 여러 파일에 중복 저장됨
- 데이터 불일치: 한 파일의 데이터를 수정했을 때 다른 파일에는 반영되지 않음
- 데이터 종속성: 파일 구조가 변경되면 이를 사용하는 애플리케이션 코드도 함께 변경되어야 함
- 여러 사용자가 동시에 동일한 파일에 접근하여 수정할 떄 충돌 제어가 어려움

## 2. 데이터베이스(Database)

___ 

### 정의 및 특징

- 정의
    - 특정 조직의 공동 업무 목절을 위해 다수의 사용자와 응용 프로그램이 공유할 수 있도록 `통합 및 저장한 데이터의 집합`
- 4대 필수 요건
    - `통합데이터`:
        - 똑같은 데이터가 여러 곳에 중복저장되는 것을 방지
        - 한 곳에 모아서 합쳐놓은 상태
    - `저장데이터`
        - 컴퓨터가 읽고 쓸 수 있는 HDD나 SSD 같은 디지털 저장 매체에 보관된 데이터
    - `운영데이터`
        - 조직이나 기업이 고유한 핵심 업무를 처리하는데 반드시 필요한 데이터
    - `공용데이터`
        - 특정인이나 하나의 프로그램이 독점하지 않음
        - 여러 사용자나 업무 시스템들이 다 함꼐 공유하여 사용할 수 있는 데이터
- 4대 특징
    - `실시간 접근성`:
        - 사용자가 데이터를 조회했을 때 오랜 시간 대기 없이 즉작적으로 빠른 결과를 받아 볼 수 있어야 함
    - `계속적인 변화`:
        - 새로운 정보가 생기면 추가
        - 틀린 내용은 수정
        - 불필요한 데이터는 삭제
        - 항 상 최신의 올바른 정보를 유지
    - `동시 공용`:
        - 여러 사용자가 같은 시점에 접속하여 동일한 데이터를 조회하거나 사용하더라도 서로 충돌없이 안전하게 이용할 수 있음
    - `내용에 의한 참조`:
        - 데이터가 디스크의 어느 주소나 물리적 위치에 보관되어 있는지 몰라도 찾고자 하는 데이터의 값을 기준으로 직접 검색할 수 있음

### DBMS (Database Management System)

- 데이터베이스를 직접 관리하고 사용자의 요구에 따라 데이터를 조회, 삽입, 수정, 삭제하는 소프트웨어
- 응용 프로그램과 물리적인 데이터 저장소 사이에서 중재자 역할을 수행
- 데이터 무결성 유지, 보안제어, 동시성 제어, 장애 복구 등의 핵심 기능을 제공

### 데이터베이스의 유형 및 비교

#### 1) 관계형 데이터베이스 (RDBMS)

- 특징
    - 데이터를 행렬로 구성된 2차원 테이블에 맞추어 정형화하여 관리
    - 사전에 컬럼 이름과 데이터 타입을 규정하는 스키마를 엄격히 정의해야 함
- 장점
    - 데이터의 무결성과 일관성이 최우선으로 보장됨
    - 여러 테이블을 조인하여 복잡한 데이터를 모순없이 다루기에 적합
- 단점
    - 데이터 저장 구조를 도중에 바꾸기 어려움
    - 대용량의 트래픽을 처리하기 위해 시스템을 여러 대로 확장(수평확장)하는 비용이 큼
- 적합한 서비스
    - 금융 시스템, 회원정보관리, 주문 및 결제 시스템 등
    - 데이터 정합성과 신뢰성이 중요한 서비스
- 대표 제품
    - MySQL, Oracle Database, PostgreSQL, Microsoft SQL Sever

#### 2) 비관계형 데이터베이스 (NoSQL)

- 특징
    - 정해진 표 구조에 얽매이지 않고 자유로운 형태로 데이터를 보관
- 장점
    - 고정된 구조가 없 어 데이터가 시시각각 변해도 쉽게 추가 가능
    - 서버를 분산하여 대규모 트래픽과 데이터를 빠르게 처리 가능
- 단점
    - 여러 데이터를 조합(JOIN)하는 연산이 매우 제한적이거나 무거움
    - 데이터 일관성과 정합성 보장이 상대적으로 약함
    - 신뢰정과 엄격한 트랜잭션관리가 요구되는 핵심 데이터 관리에는 부적합할 수 있음
- 적합한 서비스
    - 실시간 메신저(채팅), 로그 수집 시스템, 소셜 미디어(SNS) 피드 등
    - 비정형 데이터를 대량으로 신속하게 처리해야하는 서비스
- 대표 제품
    - MongoDB, Redis, Cassandra, Neo4j

#### 3) NewSQL 및 특수 목적 데이터베이스

- NewSQL: 관계형 데이터베이스의 장점과 NoSQL의 장점을 결합한 차세대 데이터 베이스
    - 장점: ACID트랜잭션 보장, SQL지원, 대량 데이터에 대한 유연한 수평 확장
    - 대표제품: Google Spanner, CockroachDB
- 특수 목적 데이터베이스
    - 벡터 데이터베이스(Vector DB): 생성형 AI(LLM)와 유사도 검색에 특화되어 대용량 고차원 백터 데이터를 다룸
        - 대표 제품: Pinecone, Milvus
    - 시계열 데이터베이스(TSDB): 서버 모니터링 로그나 센서데이터처럼 시간 흐름에 따라 순차적으로 발생하는 대량의 정보를 효율적으로 기록하고 분석
        - 대표 제품: InfluxDB, Prometheus

## 관계형 데이터베이스(RDBMS)의 개념 및 구조

- 현대 백엔드 개발에서 가장 핵심이 되는 모델
- 모든 데이터를 2차원 표(Table)형태로 저장하고 관리

### 데이터베이스, 테이블, Row, Column 구조

- `데이터베이스`: RDBMS 내에서 연관된 테이블들을 하나로 묶어 관리하는 논리적 보관 공간(스키마)
- `테이블`: 행과 열로 구성된 2차원 표 구조, 데이터를 저장하는 가장 기본적인 객체
- `행`(Row, Record):
    - 테이블에서 가로 방향의 한 줄
    - 데이터베이스에 저장되는 하나의 데이터 단위
- `열`(Column):
    - 테이블에서 세로 방향의 한 칸
    - 저장되는 데이터의 구체적인 종류와 타입을 규정
- `스키마`(Schema):
    - 테이블 구조와 데이터 타입, 테이블 간의 제약 조건에 대한 명세를 정의

### 기본 키(PK)와 외래 키 (FK)

- `기본 키`(Primary Key)
    - 테이블에서 각 행을 식별할 수 있은 유일한 값을 가지는 컬럼
    - 기본 키는 null 값을 가질 수 없으며 중복된 값을 가져서도 안됨
- `외래 키`(Foreign Key)
    - 다른 테이블의 기본 키를 참조하여 테이블 간의 유기적인 관계를 형성하는 컬럼
    - 참조 무결성 규칙에 따라 외래 키의 값은 참조 대상 테이블의 기본 키 값 중 하나이거나 null이어야 함

#### 테이블 간의 관계 (1:N, N:M)

- 1:N(일대다)관계
    - 한 쪽 테이블의 레코드 1개가 다른 쪽 테이블의 레코드 여러 개와 연결될 수 있는 가장 일반적인 관계
    - 설계 시 N에 해당하는 테이블에 외래 키(FK)를 배치하여 1에 해당하는 테이블의 기본 키(Pk)를 참조하도록 설계함
- N:M(다대다)관계
    - 양 쪽 테이블의 레코드가 서로의 레코드 여러 개와 연결될 수 있는 관계
    - RDBMS에서는 다대다 관계를 직접 테이블로 생성할 수 없음
    - 두 테이블의 기본 키를 외래 키로 가지는 매핑 테이블을 중간에 두어 1:N 및 M:1 관계로 해소하여 설계
